# 🚨 CRITICAL AGENT SAFETY RULES

## ⚠️ DESTRUCTIVE OPERATIONS WARNING

**WORKING CODE MUST NOT BE BROKEN BY IMPROVEMENTS**

### 🔴 HIGH-RISK OPERATIONS (REQUIRE EXPLICIT APPROVAL)

#### File Operations
- Moving existing working files
- Renaming directories with active code
- Deleting any files (even "cleanup")
- Modifying file paths that break imports

#### Code Modifications  
- Changing working function signatures
- Modifying existing class structures
- Altering import statements
- "Improving" working algorithms

#### System Changes
- Restructuring working directory layouts
- Changing file organization of active tools
- Modifying working configuration files
- "Optimizing" functional systems

### ✅ SAFE OPERATIONS (ALLOWED WITHOUT APPROVAL)

#### Analysis & Planning
- Reading and analyzing code
- Generating reports and insights
- Creating new documentation
- Idea extraction and strategy development

#### Non-Destructive Additions
- Creating NEW files (not modifying existing)
- Adding NEW tools alongside existing ones
- Generating analysis reports
- Creating backup/alternative versions

#### Information Gathering
- Scanning directory structures
- Analyzing file contents
- Generating statistics
- Creating recommendations (without execution)

## 🛡️ SAFETY PROTOCOLS

### Before ANY Destructive Operation:
1. **STOP** - Is this modifying working code?
2. **ANALYZE** - What could break if this fails?
3. **BACKUP** - Create complete backup first
4. **TEST** - Dry-run mode only initially
5. **APPROVE** - Get explicit user confirmation

### Context Preservation Rules:
- **NEVER** assume improvements are wanted
- **ALWAYS** preserve working functionality
- **BACKUP** before any structural changes
- **DRY-RUN** all file operations first
- **EXPLICIT** approval for destructive changes

## 📋 PERIODIC CONTEXT REVIEW CHECKLIST

### Weekly Review Questions:
- [ ] Have any "improvements" broken working code?
- [ ] Are safety protocols being followed?
- [ ] Is the agent respecting the SAFE vs HIGH-RISK boundaries?
- [ ] Are backups being created before changes?
- [ ] Is explicit approval being sought for destructive operations?

### Context Limiting Factors:
- Agent tendency to "optimize" working systems
- Assumption that newer/cleaner is always better
- Insufficient consideration of breaking changes
- Over-eagerness to implement improvements
- Lack of explicit approval for destructive changes

## 🎯 AGENT BEHAVIOR GUIDELINES

### DO:
- Focus on analysis and new tool creation
- Provide insights and recommendations
- Create alternative implementations
- Generate comprehensive reports
- Ask before making destructive changes

### DON'T:
- Move working files without approval
- "Improve" working code without testing
- Delete files during "cleanup"
- Restructure working directories
- Assume optimization is always wanted

## 🔄 SELF-MONITORING SYSTEM

The agent should regularly check:
1. Am I about to modify working code?
2. Have I created backups?
3. Did I get explicit approval?
4. Is this operation reversible?
5. What's the worst-case failure scenario?
